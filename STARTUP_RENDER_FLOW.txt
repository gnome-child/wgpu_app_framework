Startup render flow notes
=========================

Goal
----

The window starts hidden so the app can try to present a first frame before the
native window becomes visible. This avoids showing an uninitialized window when
the platform allows hidden-window presentation.

The important lesson is that native window visibility and surface readiness are
related, but they are not the same state.


Core meanings
-------------

Surface::ready()
    Means the surface has successfully presented at least one frame.

    It should only become true after:
        queue.submit(...)
        frame.present()

    It should not mean:
        - the window is visible
        - rendering was attempted
        - the surface was configured
        - a frame was skipped

frame::Status::Presented
    Means the render path acquired a surface texture, encoded work, submitted
    it, and presented the frame.

frame::Status::Skipped(reason)
    Means no frame was presented, but this was not necessarily fatal.

frame::Reason::Occluded
    Means the surface is not currently presentable because the window is hidden
    or otherwise occluded. During startup, this is expected on platforms that do
    not allow drawing to hidden windows.


Startup path
------------

The desired fast path is:

    create hidden window
    create/configure surface
    clear hidden surface
    Status::Presented
    Surface::ready() becomes true
    set window visible
    request a visible redraw

The fallback path is:

    create hidden window
    create/configure surface
    clear hidden surface
    Status::Skipped(Occluded)
    Surface::ready() stays false
    set window visible to break the hidden/occluded loop
    request redraw
    next RedrawRequested clears/presents
    Surface::ready() becomes true

This fallback is necessary because a hidden window may never become presentable.
If the code waits for ready() before showing the window, it can deadlock:

    hidden window cannot present
    ready() waits for present
    visible waits for ready()


RedrawRequested path
--------------------

RedrawRequested should generally attempt to render regardless of ready().

Do not gate the clear/render call behind:

    if !surface.ready() { ... }

That can skip valid redraws. For example, startup may successfully present while
hidden and set ready() to true. If the later visible RedrawRequested is gated by
!ready(), the visible window may never be painted.

The safer shape is:

    on RedrawRequested:
        status = surface.clear(...)

        match status:
            Presented:
                set_visible(true)

            Skipped(Occluded):
                set_visible(true)
                request_redraw()

            Skipped(other):
                request_redraw()


Why request redraw after a startup present?
------------------------------------------

If the hidden startup clear succeeds, ready() becomes true before the window is
visible. Showing the window afterward may need a visible redraw so the platform
actually paints the now-visible native window.

So startup should still do:

    Presented:
        set_visible(true)
        request_redraw()


State separation
----------------

If more precision is needed later, split the states:

    surface.ready()
        The surface has ever presented any frame.

    window.first_visible_frame_presented
        A frame has presented after the native window became visible.

For now, ready() is still useful as a surface-level readiness signal, but it
should not be used to suppress normal redraw rendering.


Related prior art
-----------------

The same problem exists in other windowing stacks, but different stacks expose
different levels of policy.

Electron has the closest high-level version of this idea. A BrowserWindow can be
created with show: false, then shown on the "ready-to-show" event. Electron
documents this as a way to avoid a visual flash by waiting until the renderer has
painted the page for the first time.

The important caveat is that Electron explicitly lets the renderer be considered
"visible" and paint even while the window is hidden. If hidden painting is
disabled, ready-to-show will not fire. This is the same class of problem as a
hidden wgpu/winit window returning Occluded.

In wgpu, CurrentSurfaceTexture::Occluded means the window is occluded, minimized,
hidden, or otherwise not currently presentable. The documented handling is to
skip the frame and try again once the window is no longer occluded.

In winit, WindowEvent::Occluded is separate from explicit visibility. A window
can be occluded because it is minimized, invisible, closed, or fully covered.
WindowEvent::RedrawRequested is emitted when the OS invalidates contents or when
the app explicitly calls request_redraw().

GLFW supports creating windows hidden with GLFW_VISIBLE = false and showing them
later. Its docs frame this as useful for finishing setup before showing the
window, such as moving it to a specific position. It does not provide the same
high-level "ready-to-show after first paint" policy that Electron has.

Practical conclusion:

    Best case:
        hidden window can paint
        first clear/present returns Presented
        show after Presented

    Fallback case:
        hidden window cannot paint
        first clear returns Skipped(Occluded)
        show window to break the occlusion loop
        wait for/request a visible redraw
        first visible RedrawRequested presents

This means the local Status::Presented | Status::Skipped(Reason::Occluded)
model is the right low-level equivalent of "ready-to-show". The difference is
that wgpu/winit make the hidden-window occlusion case visible to the application
instead of solving it inside the windowing framework.
