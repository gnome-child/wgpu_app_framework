Architecture Audit Follow-Ups
=============================

Widget extension boundary
-------------------------
- Keep ui as the retained tree, event, layout, and painting substrate.
- Keep concrete desktop components in widget.
- Define the eventual extension point for custom widgets before adding many one-off widget helpers.
- Avoid moving widget-owned metrics, parts, popup construction, or chrome into app/runtime again.

Pointer capture ownership
-------------------------
- pointer::Capture is currently a value type with runtime-owned lifecycle.
- Before adding split panes, sliders, drag/drop, or text selection, centralize capture policies for widget parts, action controls, and future document interactions.
- Make sure capture can express pressed visual state without making pointer state widget-owned truth.

Generated IDs
-------------
- Menu rows, submenu rows, scroll chrome, and future generated widget parts need stable generated ids.
- Add a small generated-id helper so paths stay predictable and do not rely on repeated ad hoc string formats.
- Keep generated ids clearly outside the app-authored id namespace where practical.

Theme derivation
----------------
- Continue deriving child chrome from parent geometry and density tokens instead of copying magic numbers.
- Floating panels should remain the source for popup/menu glass, shadow, stroke, padding, and rounding.
- Menu row geometry, separator placement, popup insets, and gutter/corner sizing should be derived from shared metrics.
- Avoid separate style tokens when the value can be computed from a parent surface token.

Render geometry helpers
-----------------------
- Analytic shapes, shadows, outlines, clips, and backdrop masks all carry similar snapped rect, raster bounds, and resolved rounding data.
- Factor shared geometry preparation into one private render helper before adding transforms, shadows variants, or clipping refinements.
- Keep shape math reusable by blur, shadow, clip, and future image/material passes.

Paint constructors and accessors
--------------------------------
- Several paint data types are still easy to construct directly.
- Move paint toward private fields plus constructors/accessors where invariants matter, matching the mp4 crate style.
- Keep paint concrete and backend-neutral; avoid reintroducing generic effect objects unless a real abstraction appears.

Command subject naming cleanup
------------------------------
- Public APIs still expose command_target compatibility names.
- Internally the model is command subject plus scoped capture.
- Rename public command_target methods only after menus, command runner, shortcuts, and text responders settle enough to justify a breaking cleanup.

Text measurement quantization
-----------------------------
- text::Measurer now caches shaped metrics, but cache keys include exact bounds and style values.
- Quantize or normalize measurement bounds where practical before adding large lists, inspectors, or editors.
- Keep render-side glyph buffer reuse separate from measurement caching.

Future widget gaps
------------------
- Needed soon: text field/editor, list/table, split pane, toolbar, command runner, context menu, checkbox/toggle, slider, scrollbar keyboard behavior, and scrollbars with optional overlay style.
- Add widgets through shared composition, metrics, pointer capture, theme, and action/responder paths.
- Do not let each widget invent its own hit testing, sizing, generated ids, or state ownership model.
